### 1. Почему capacity >= size? По какому принципу растут?

size это сколько элементов сейчас в векторе, capacity сколько памяти под них выделено.
capacity >= size потому что вектор заранее берет память с запасом, иначе при каждом push_back пришлось бы делать realloc и копировать всё заново

из результатов видно что capacity всегда степень двойки: при size=257 capacity=512, при size=513 capacity=1024 и тд. то есть когда место кончается все удваивается, из-за этого push_back дорогой только иногда, в среднем константная сложнсость

### 2. Как меняется время при увеличении N? Асимптотика?

random_access вообще не меняется — 50нс при N=256 и 38нс при N=32768, это O(1), всё логично

push_back тоже почти не растет, хотя при N=8192 был скачок до 2265нс

push_front и random_insert растут заметно: push_front был 369нс при 256 элементах, стал 5311нс при 32768. это примерно x14, что похоже на линейный рост. random_erase ведет себя похоже

### 3. Какие операции самые дорогие?

push_front — вставка в начало сдвигает все элементы на одну позицию вправо. при 32768 элементах это 32768 сдвигов за одну операцию.

random_insert примерно так же, зависит от позиции (в коде вставляю на позицию 9, поэтому сдвигается почти всё).

random_erase чуть быстрее чем insert судя по числам, хотя логика та же — просто удаление может быть немного дешевле чем вставка, не уверен почему именно.

### 4. Для чего reserve?

если знаешь заранее сколько элементов добавишь — лучше сразу выделить нужную память через reserve. тогда не будет промежуточных реаллокаций.

я делаю shrink_to_fit перед каждым замером чтобы capacity была ровно равна size и замер был "честным"

### 5. Для чего shrink_to_fit?

если долго наполнял вектор а потом удалил кучу элементов, capacity не уменьшается сама. shrink_to_fit говорит: верни лишнюю память. это hint а не гарантия, но обычно работает

### 6. Почему вектор передают по ссылке, в т.ч. const?

при передаче по значению весь вектор копируется — это O(N) и лишняя память. по const& передается фактически указатель, никакого копирования. const нужен чтобы случайно не изменить вектор внутри функции

### 7. Отличие от динамического массива? От статического? Есть яли статический массив в STL?

динамический массив (new int[n]) — нужно самому помнить про delete[], нет .size(), не работает с алгоритмами stl нормально. вектор всё это берет на себя.

статический массив (int arr[N]) — живет на стеке, размер фиксирован на этапе компиляции. вектор на куче, может расти.

в stl есть std::array<T, N> — это обертка над статическим массивом. размер шаблонный параметр, данные на стеке. удобнее голого массива тем что знает свой размер и работает с range-for
