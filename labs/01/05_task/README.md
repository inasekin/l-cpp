# Задание 5

## Почему нельзя просто написать void f(int arr[][]), если размеры неизвестны на этапе компиляции?

Потому что компилятору необходимо знать размеры массивов. При передаче многомерного массива в функцию как int arr[][] компилятор не сможет определить смещения для доступа к элементам

## Чем «зубчатый» массив (int**) отличается от настоящего двумерного массива (int[rows][cols]) в памяти?

Основное отличие в непрерывности памяти и способе доступа.

Двумерный массив : эанимает единый, непрерывный блок памяти. Элементы расположены последовательно друг за другом. Доступ к элементу [i][j] осуществляется по формуле с известным количеством столбцов, что быстро и эффективно.

«Зубчатый» массив: это массив указателей, где каждый указатель ведет на начало отдельного одномерного массива. Строки не обязаны находиться в непрерывном блоке памяти, они могут быть разной длины и располагаться хаотично в куче 

## Почему при освобождении нужно сначала удалять каждую строку, а потом массив указателей?

Из-за порядка выделения памяти и иерархии. При создании «зубчатого» массива память выделяется в два этапа: сначала выделяется память под массив указателей (int**), а затем в цикле память под каждую отдельную строку.

## Что произойдёт, если вызвать delete[] только на matrix, но не на matrix[i]?

Произойдет memory leak. Будет освобожден только верхний уровень массив указателей

## Почему в resizeMatrix используется int**& matrix? Можно ли обойтись без ссылки?

Ссылка используется, чтобы изменить сам указатель matrix в вызывающей функции

Можно ли обойтись без ссылки? Да, можно. Функция могла бы просто вернуть новый указатель на новую матрицу.

## Какие риски связаны с использованием int**?

Использование int** в C++ сопряжено с несколькими серьезными рисками:

- Утечки памяти. Ручное управление памятью (new[]/delete[]) подвержено ошибкам. Забытый delete[] приводит к утечкам;
- Небезопасный доступ. Отсутствие проверок границ массива может привести к выходу за пределы выделенной памяти (buffer overflow) и неопределенному поведению программы
- Сложность владения. Не всегда сразу понятно, какая часть кода отвечает за освобождение памяти, особенно при передаче указателей между функциями; 
- Фрагментация памяти. Выделение множества небольших, несвязанных блоков памяти может привести к неэффективному использованию кучи;
